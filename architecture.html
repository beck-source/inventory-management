Request Data Flow
1
User changes a filter
FilterBar.vue updates a shared ref in the useFilters composable. A watch() in the active view fires.
selectedLocation.value = 'Tokyo' // reactive ref, shared across all views
2
api.js builds an HTTP request
The view calls the relevant method in api.js, which serializes filter state into query parameters.
GET /api/orders?warehouse=Tokyo&category=all&status=all&month=all
3
FastAPI filters in-memory data
apply_filters() chains warehouse, category, and status filters over the Python list loaded at startup. filter_by_month() expands quarter strings to month lists before matching.
apply_filters(orders, warehouse='Tokyo', category='all', status='all') # → filter_by_month(result, month='all') ← expands Q1-2025 → ['2025-01','2025-02','2025-03']
4
Pydantic validates & serializes
Response objects are validated against Pydantic models before JSON serialization, enforcing schema consistency at the boundary.
class Order(BaseModel): id: str; sku: str; status: str; warehouse: str; total_value: float ...
5
Vue updates computed properties
Raw data lands in a ref. Derived views (sorted tables, aggregated KPIs) live in computed() and re-render automatically. Search is always client-side over already-fetched data.
const items = ref([]) // raw API response const filteredItems = computed(() => // derived — no extra fetch items.value.filter(i => i.name.includes(searchQuery.value)) )

Filter System
4 global filters live in useFilters.js as shared reactive refs. FilterBar updates them; every view watches them and re-fetches.
Time Period
month · quarter · YTD · all
Warehouse
SF · Chicago · Tokyo · all
Category
product category · all
Order Status
Pending · Delivered · all
Search is always client-side Inventory name search filters the already-fetched items via a computed property — no extra API call. Backend filters handle all other dimensions.

State & Reactivity Patterns
Composable-based state (no Vuex/Pinia)
useFilters() is a singleton — calling it from any component returns the same reactive refs. FilterBar writes, views read and watch. No global store needed.
Raw ref + computed pattern
Each view holds raw API data in a ref. Sorted, filtered, and aggregated views are computed() — zero duplication, automatic invalidation.
