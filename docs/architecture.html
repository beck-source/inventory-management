Request Data Flow

1

User changes a filter

FilterBar.vue updates selectedPeriod / selectedLocation / selectedCategory / selectedStatus refs inside the useFilters composable (module-level singleton — shared across all views).

2

View watches filters and re-fetches

Each view calls getCurrentFilters() which maps selectedPeriod → month and selectedLocation → warehouse, then passes the result to an api.js method (e.g. api.getOrders(filters)).

3

api.js builds the HTTP request

Axios encodes filters as URLSearchParams. Any filter value equal to "all" is silently omitted, keeping URLs clean. Example: GET /api/orders?warehouse=Chicago&status=Processing

4

FastAPI filters in-memory data

apply_filters() iterates the in-memory list and matches on warehouse, category, and status. filter_by_month() then handles direct month strings (2025-09) or quarters (Q3-2025) via QUARTER_MAP.

5

Pydantic validates & serialises

FastAPI validates the filtered list against the declared response_model. The response is JSON-serialised and returned to the client.

6

Vue renders via computed properties

Raw API responses are stored in ref()s (allOrders, inventoryItems, …). Derived values (totals, charts, formatted values) live in computed() properties that re-evaluate automatically on change.

Global Filter System

Four filters in useFilters.js are shared as module-level refs — changing one filter instantly triggers re-fetch in every active view.

API Endpoints

GET /api/inventory All inventory items ?warehouse &category
GET /api/inventory/{id} Single inventory item
GET /api/orders All orders ?warehouse &category &status &month
GET /api/orders/{id} Single order
GET /api/dashboard/summary KPI totals — inventory value, low stock, pending orders, backlog count ?warehouse &category &status &month
GET /api/demand Demand forecasts with trend (stable / increasing / decreasing)
GET /api/backlog Backlog items — enriched with has_purchase_order flag
GET /api/spending/summary Budget vs actual spending summary
GET /api/spending/monthly Month-by-month spending breakdown
GET /api/spending/categories Spending by product category
GET /api/spending/transactions Recent purchase transactions
GET /api/reports/quarterly Quarterly order counts, revenue & fulfillment rate (computed from orders)
GET /api/reports/monthly-trends Month-over-month order count, revenue & delivery counts
POST /api/purchase-orders Create a purchase order for a backlog item
GET /api/purchase-orders/{backlog_item_id} Get PO for a specific backlog item
GET /api/tasks Task list (in-memory, resets on restart)
POST /api/tasks Create task
PATCH /api/tasks/{id} Toggle task completion
DELETE /api/tasks/{id} Delete task

Data Layer — server/data/

All JSON files are loaded once at server startup into module-level globals in mock_data.py. Changes to JSON require a server restart. Data covers 12 months across 4 product categories and multiple warehouse locations.

inventory.json

SKUs, quantity on hand, reorder points, unit costs, warehouse locations

orders.json

Customer orders with status, dates, total value, warehouse & category tags

demand_forecasts.json

Per-SKU current vs forecasted demand with trend classification

backlog_items.json

Delayed order lines with quantity shortfall, days delayed, and priority

purchase_orders.json

Supplier POs linked to backlog items by backlog_item_id

spending.json

Composite file: spending_summary, monthly_spending, category_spending

transactions.json

Recent individual purchase transactions for the spending view
