Filter Data Flow

Step 1
User Selects Filter
FilterBar.vue renders 4 dropdowns. v-model binds to shared composable state.
selectedLocation = "San Francisco"
selectedStatus = "delivered"
selectedPeriod = "2025-01"
Step 2
Composable Normalizes
useFilters.getCurrentFilters() maps UI values to API param names. "all" values are dropped.
{ warehouse: "San Francisco",
  status: "delivered",
  month: "2025-01" }
Step 3
API Call via Axios
api.js builds URLSearchParams and fires GET request. Watch in the view triggers on filter change.
GET /api/orders
  ?warehouse=San+Francisco
  &status=delivered
  &month=2025-01
Step 4
Server Filters & Returns
apply_filters() and filter_by_month() reduce the in-memory dataset. Pydantic validates output.
apply_filters(orders,
  warehouse, category, status)
filter_by_month(result, month)
→ List[Order] (Pydantic)

Key Architectural Patterns

Singleton Filter State
useFilters.js is imported by all views and FilterBar. Because module state is shared, all components see the same reactive filter values without a store like Pinia/Vuex.
Watch-Triggered API Calls
Each view watches the filter composable and re-fetches from the API whenever a filter changes. Raw server data lands in a ref; computed properties derive display-ready values from it.
Server-Side Filtering
apply_filters() and filter_by_month() run in the FastAPI layer, not the browser. The frontend receives only the records it needs rather than downloading the full dataset.
Pydantic Response Validation
Every endpoint return type is a typed Pydantic model. FastAPI serializes and validates automatically — the frontend can rely on shape guarantees without defensive null checks.
In-Memory Mock Data
JSON files are loaded once at startup by mock_data.py and held in module-level variables. No database, no ORM — mutations during a session are lost on restart.
Locale-Driven Currency
useI18n composable stores locale in localStorage. Switching EN↔JA automatically changes currency formatting (USD↔JPY) and all translated strings across the app.
