System Architecture

Factory Inventory Management System

A demo full-stack application for tracking factory inventory, orders, demand forecasting, backlog, and spending. Three-tier architecture with no database — all data is loaded into memory from JSON files at startup.

Frontend Vue 3 · Vite · :3000 Backend FastAPI · Uvicorn · :8001 Data 7 JSON datasets, in-memory Tests pytest + FastAPI TestClient

01System Architecture

A classic three-tier layout: a Vue single-page app, a FastAPI HTTP service, and an in-memory data layer hydrated from flat JSON files. There is no persistence — restarting the server resets all state.

03Data Flow — Request Lifecycle

Every page follows the same loop: a global filter bar drives shared state, views fetch filtered data from the API, and the backend filters its in-memory lists before returning Pydantic-validated JSON.

1

User changes a filter

FilterBar.vue exposes Time Period, Warehouse, Category, and Order Status. Selections are written to singleton refs in composables/useFilters.js, so every view observes the same filter state.

2

View reacts and calls the API client

A watch on the active view (e.g. views/Dashboard.vue) fires and calls api.js, which appends only non-all filters as query params and issues an axios GET.

3

FastAPI route filters in memory

The route handler in server/main.py receives optional query params, then filters the global lists from mock_data.py with helpers like filter_by_month() (which also expands quarters such as Q3-2025).

4

Pydantic validates and serialises the response

Endpoints declare response_model (e.g. List[InventoryItem], Order, BacklogItem). Pydantic guarantees the JSON shape, so the frontend can assume field presence without defensive checks.

5

View stores raw data, derives via computed

Raw arrays land in refs (e.g. allOrders, inventoryItems). KPIs, chart series, and table rows are exposed as computed properties so the UI stays reactive when filters change again.

6

Render — SVG charts, tables, modals

Views render hand-built SVG charts and CSS-grid tables. Detail modals (InventoryDetailModal, BacklogDetailModal, …) open on row click and may issue follow-up requests like POST /api/purchase-orders.

06Design Notes & Constraints

No database, no persistence

All data is module-level Python lists loaded at import time. Mutations (e.g. creating a purchase order) survive only until the process restarts. Demo-friendly, not production-ready.

No state library

Cross-page state (filters, auth, locale) is handled with the singleton-composable pattern: module-scoped refs exported from composables/. Simpler than Pinia for an app of this size.

Hand-rolled SVG charts

Charts are plain SVG generated in templates and computed props — no charting library. Keeps the bundle small and the visuals on-brand with the slate design system.

Filter dimensions are uneven

Orders support all four filters; inventory has no time dimension so it ignores month; demand, backlog, and spending take none. The frontend only appends non-all params.

Contract enforced by Pydantic

Each endpoint declares a response_model, so changing a JSON data file requires updating the matching Pydantic class in main.py or requests will 500.

i18n without a library

useI18n swaps between locales/en.js and locales/ja.js with a reactive t() helper. Currency formatting lives in utils/currency.js.
